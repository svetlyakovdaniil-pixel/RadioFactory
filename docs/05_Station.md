# Station

## 1. Назначение

`Station` — главная автономная рабочая единица RadioFactory. Каждая Station представляет отдельную круглосуточную радиостанцию внутри одного YouTube Channel.

Примеры: Hip-Hop, Lo-Fi, Jazz, Chill, Deep Space, Ambient.

Один YouTube Channel может содержать несколько Station. Каждая ведёт собственный LIVE и не зависит от внутреннего состояния других Station.

## 2. Место в архитектуре

```text
Workspace
└── YouTube Channel
    ├── Station A
    │   ├── Configuration
    │   ├── Media Library
    │   ├── Pending LIVE Package
    │   ├── Runtime State
    │   ├── Broadcast History
    │   └── Events and Logs
    └── Station B
```

Station принадлежит одному Workspace и одному YouTube Channel.

## 3. Автономность

Каждая Station имеет собственные:

- настройки;
- медиатеку;
- музыкальную очередь;
- loop-видео;
- превью;
- названия и описания;
- состояние;
- активный Broadcast;
- Pending LIVE Package;
- историю;
- события и ошибки;
- процедуру восстановления.

Сбой одной Station не должен останавливать другие Station, менять их медиатеки, сбрасывать их очереди, очищать их временные данные или блокировать общую очередь восстановления.

Исключение допускается только для подтверждённой общей причины уровня Workspace.

## 4. Идентичность

У Station должен быть стабильный внутренний идентификатор, независимый от отображаемого названия.

Переименование не должно менять идентификатор, терять историю, сбрасывать настройки, создавать новую Station или разрывать связь с активным Broadcast.

Формат идентификатора пока не утверждён.

## 5. Создание

Station создаётся через пошаговый мастер.

Подтверждено:

- шаблоны не используются;
- клонирование не используется;
- обязательные шаги нельзя пропустить;
- Station не считается готовой, пока не заполнены обязательные параметры.

Мастер должен привести пользователя к определению YouTube Channel, названия Station, обязательных ресурсов, настроек LIVE и параметров публикации. Точный порядок шагов ещё не утверждён.

## 6. Данные Station

### 6.1 Configuration

Конфигурация хранится отдельно от runtime-state и включает как минимум:

- отображаемое название;
- принадлежность YouTube Channel;
- целевую длительность LIVE;
- параметры выбора ресурсов;
- настройки публикации;
- выбранный YouTube-плейлист.

### 6.2 Media Library

Station владеет отдельными наборами музыки, loop-видео, превью, названий и описаний.

Ресурс доступен LIVE только после успешной обработки и проверки.

### 6.3 Runtime State

Runtime-state должен позволять восстановить:

- была ли Station запущена пользователем;
- текущий статус;
- активный Broadcast;
- текущий трек и позицию;
- музыкальную очередь;
- Pending LIVE Package;
- выполняемую команду;
- последнюю ошибку;
- необходимость восстановления;
- время последнего heartbeat.

### 6.4 History

История связана со стабильным идентификатором и сохраняется после переименования, перезапуска, завершения LIVE и временной ошибки.

## 7. Готовность к запуску

Минимальные обязательные условия:

- один обработанный музыкальный трек;
- одно обработанное loop-видео;
- одно превью;
- одно название;
- одно описание;
- рабочее подключение к YouTube;
- обязательные настройки публикации.

Отсутствие ресурса — состояние ожидания, а не Error.

Подтверждённые состояния:

- `Waiting for Music`;
- `Waiting for Loop Video`;
- `Waiting for Thumbnail`;
- `Waiting for Titles`;
- `Waiting for Descriptions`.

После появления ресурса Station автоматически продолжает подготовку.

## 8. Пользовательское намерение

RadioFactory различает:

- Station, остановленную вручную;
- Station, которая должна работать, но временно не работает из-за сбоя.

После перезапуска вручную остановленная Station остаётся остановленной. Ранее работавшая Station добавляется в очередь восстановления.

## 9. Управление

Подтверждённые команды:

- запуск;
- штатная остановка;
- принудительная остановка;
- открытие детальной страницы;
- управление медиатекой;
- изменение настроек.

Dashboard не управляет FFmpeg напрямую. Команда проходит через backend и исполняется компонентом Station.

## 10. Запуск

Запуск — многоэтапная операция.

Пользователь должен видеть:

- что команда принята;
- текущий этап;
- ожидание ресурса;
- понятную ошибку;
- успешный переход в LIVE.

Новый Broadcast нельзя создавать, пока предыдущий безопасно не завершён, Cleanup не окончен или уже выполняется конфликтующая операция.

## 11. Штатная остановка

`Остановить`:

1. прекращает запуск новых треков;
2. даёт текущему треку доиграть;
3. корректно завершает LIVE;
4. выполняет Cleanup;
5. переводит Station в остановленное состояние;
6. сохраняет ручное намерение остановки.

## 12. Принудительная остановка

`Остановить принудительно`:

1. требует подтверждения `Остановить / Отмена`;
2. может немедленно прервать текущий трек;
3. быстро завершает LIVE;
4. выполняет максимально безопасный Cleanup;
5. сохраняет Station как вручную остановленную.

Команда должна визуально отличаться от штатной остановки.

## 13. Ошибки

Station переходит в Error только когда работа не может безопасно продолжаться и автоматические попытки исчерпаны.

Error одной Station не останавливает остальные.

Сообщение имеет два уровня:

1. понятное объяснение и требуемое действие;
2. код ошибки и технические сведения.

## 14. Предупреждения

Предупреждение не останавливает Station.

Оно:

- отображается в Dashboard;
- отправляется в Telegram;
- содержит рекомендацию;
- не заменяет основной статус.

## 15. Автоматическое восстановление

RadioFactory сначала использует безопасные автоматические способы восстановления.

Пользователь привлекается только после их исчерпания или когда требуется ручное действие.

Неудача одной Station переводит только её в Error и не останавливает очередь остальных.

## 16. Восстановление после перезагрузки

1. Workspace определяет ранее работавшие Station.
2. Ручные остановки сохраняются.
3. Нужные Station попадают в очередь.
4. Восстановление идёт последовательно или ограниченными группами.
5. Учитываются ресурсы сервера.
6. Ошибка одной Station не блокирует остальные.

## 17. Недостаток места

Останавливаются только затронутые Station. Остальные продолжают работу.

После освобождения места выполняется автоматическая попытка восстановления. При окончательной неудаче Station переходит в Error.

## 18. Конфликт команд

Одна Station не должна одновременно выполнять две операции, изменяющие один runtime-state или жизненный цикл Broadcast.

Примеры конфликтов:

- повторный запуск во время запуска;
- запуск во время остановки;
- штатная и принудительная остановка одновременно;
- новый Broadcast во время Cleanup;
- изменение обязательных ресурсов во время фиксации Pending LIVE Package.

Точная модель очереди команд ещё не утверждена.

## 19. Границы ответственности

Station отвечает за собственную бизнес-логику и состояние.

Station не отвечает за глобальную аутентификацию, список всех каналов, общие лимиты Workspace, чужие настройки и прямое управление из браузера внутренними процессами.

## 20. Открытые вопросы

- формат идентификатора;
- формат конфигурации и runtime-state;
- heartbeat;
- конкурентное изменение настроек;
- удаление и архивация Station;
- перенос между YouTube-каналами;
- приоритет Station;
- временная пауза без завершения LIVE.
