# Broadcast

## 1. Назначение

`Broadcast` — отдельный экземпляр YouTube LIVE конкретной Station.

Station — долгоживущая сущность. Broadcast — ограниченный по времени эфир внутри непрерывной работы Station.

```text
Station
├── Broadcast 1 — завершён
├── Broadcast 2 — завершён
├── Broadcast 3 — активен
└── Pending LIVE Package — следующий
```

## 2. Непрерывный режим

```text
Prepare
→ Create LIVE
→ Start LIVE
→ Broadcast
→ Natural Finish
→ Cleanup
→ Prepare Next LIVE
```

Запланированное ожидание между LIVE не используется. Безопасность завершения и Cleanup важнее скорости запуска следующего LIVE.

## 3. Независимость Broadcast

Каждый новый Broadcast получает собственные:

- YouTube identifiers;
- музыкальный цикл;
- loop-видео;
- превью;
- название;
- описание;
- время старта;
- целевую длительность;
- runtime-state;
- историю событий.

Новый Broadcast не продолжает незавершённую музыкальную очередь предыдущего. Исключение — восстановление того же Broadcast после краткого разрыва.

## 4. Pending LIVE Package

До создания YouTube LIVE фиксируется неизменяемый пакет:

- музыкальная очередь;
- loop-видео;
- превью;
- название;
- описание;
- настройки публикации;
- целевая длительность;
- данные для повторной попытки.

Если создание LIVE не удалось, используется тот же пакет. Ресурсы не выбираются заново.

Новый пакет формируется только после успешного завершения предыдущего Broadcast, отмены подготовки пользователем или изменения обязательного ресурса, делающего пакет недействительным.

## 5. Минимальная готовность

Требуются:

- музыка;
- loop-видео;
- превью;
- название;
- описание;
- рабочее подключение к YouTube;
- параметры публикации.

Если ресурса нет, Station ждёт его и не создаёт Broadcast.

## 6. Музыкальный цикл

Правила:

- каждый доступный трек входит один раз;
- порядок случайный;
- повторов внутри цикла нет;
- после полного цикла создаётся новый;
- новый Broadcast начинает новый цикл;
- текущий цикл не перестраивается после изменения библиотеки;
- следующий цикл использует актуальную библиотеку.

Политика удаления файла, уже попавшего в активную очередь, пока не утверждена.

## 7. Выбор ресурсов

Для каждого Broadcast случайно выбираются loop-видео, превью, название и описание.

Если альтернативы существуют, два последовательных Broadcast одной Station не используют одинаковые варианты.

Глобальное исключение повторов между Station не утверждено.

## 8. Целевая длительность

Длительность является целевой.

- текущий трек всегда доигрывает;
- следующий запускается только если полностью помещается;
- треки не обрезаются;
- фактическая длительность может быть немного меньше целевой.

## 9. Создание YouTube LIVE

Повторное нажатие `Запустить` не должно создавать несколько LIVE.

RadioFactory должна отличать:

- команда принята;
- создание выполняется;
- LIVE создан;
- технический шаг можно повторить;
- операция окончательно не удалась.

Механизм идемпотентности будет определён в Backend/API.

## 10. Пошаговый запуск

Dashboard должен различать как минимум:

- проверку готовности;
- фиксацию Pending LIVE Package;
- создание YouTube LIVE;
- подготовку воспроизведения;
- подключение к RTMP;
- подтверждение фактического эфира.

Финальные названия этапов ещё не утверждены.

## 11. Состояние активного Broadcast

Нужно хранить и показывать:

- Station;
- YouTube LIVE;
- время старта;
- целевую и прошедшую длительность;
- текущий трек;
- позицию и остаток трека;
- музыкальную очередь;
- выбранные ресурсы;
- состояние подключения;
- heartbeat;
- предупреждения;
- последнюю ошибку.

## 12. Естественное завершение

1. Текущий трек доигрывает.
2. Новые треки не запускаются.
3. Поток корректно завершается.
4. YouTube LIVE закрывается.
5. Итог фиксируется.
6. Выполняется Cleanup.
7. Station готовит следующий Broadcast.

## 13. Штатная остановка

При `Остановить`:

- новые треки не запускаются;
- текущий трек доигрывает;
- LIVE завершается;
- выполняется Cleanup;
- следующий Broadcast не создаётся;
- Station сохраняется как вручную остановленная.

## 14. Принудительная остановка

После подтверждения:

- текущий трек может быть прерван;
- поток прекращается;
- LIVE завершается максимально быстро;
- выполняется доступный безопасный Cleanup;
- следующий Broadcast не создаётся.

## 15. Cleanup

Cleanup изолирован для каждого Broadcast.

Цели:

- завершить процессы;
- освободить временные ресурсы;
- закрыть дескрипторы;
- зафиксировать итог;
- исключить влияние старого Broadcast на новый;
- не допустить дубли процессов.

Cleanup не удаляет постоянную медиатеку, чужие данные, историю или настройки.

Точный состав ещё не утверждён.

## 16. Краткий разрыв

Подтверждённое правило:

- менее 3 минут — продолжить тот же Broadcast;
- 3 минуты или больше — завершить старый, выполнить Cleanup и создать новый.

При продолжении того же Broadcast сохраняются пакет ресурсов, музыкальный цикл и идентичность LIVE.

## 17. Продолжение текущего трека

При восстановлении менее чем через 3 минуты трек продолжается с максимально точной позиции прерывания.

Система должна знать текущий трек, позицию до сбоя, длительность разрыва и безопасную позицию продолжения.

Допустимая погрешность ещё не утверждена.

## 18. Длительный разрыв

При разрыве 3 минуты или больше:

1. старый Broadcast не продолжается;
2. его состояние завершается;
3. выполняется Cleanup;
4. формируется новый пакет;
5. создаётся новый Broadcast;
6. начинается новый музыкальный цикл.

## 19. Ошибка создания

Пока повтор безопасен, используется тот же Pending LIVE Package.

Не создаются дублирующие LIVE и не меняются ресурсы.

После исчерпания попыток Station переходит в Error, а остальные Station продолжают работу.

## 20. Изоляция

Каждый Broadcast имеет собственные:

- идентификаторы;
- runtime-state;
- процессы;
- временные файлы;
- логи;
- Cleanup;
- ошибки;
- историю.

Глобальный «текущий Broadcast» для всего Workspace недопустим.

## 21. История

История должна отвечать:

- какая Station создала LIVE;
- когда он подготовлен, начат и завершён;
- как завершён;
- какой пакет использован;
- какие ошибки произошли;
- выполнялось ли восстановление;
- была ли ручная или принудительная остановка.

## 22. Наблюдаемость

Пользователь видит понятное состояние без чтения сырых логов. Технические логи сохраняются для диагностики.

## 23. Открытые вопросы

- точный список этапов запуска;
- состав Cleanup;
- количество повторов по типам ошибок;
- допустимая погрешность восстановления;
- удаление активного трека;
- момент публикации превью;
- обновление метаданных;
- зависший, но формально активный LIVE;
- ограничения YouTube;
- формат истории и runtime-state;
- heartbeat;
- изменение настроек во время LIVE.
