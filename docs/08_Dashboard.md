# Dashboard

## 1. Назначение

Dashboard — основной пользовательский интерфейс RadioFactory.

Он позволяет:

- видеть состояние Workspace;
- контролировать все YouTube-каналы и Station;
- замечать проблемы;
- управлять Station;
- работать с медиатекой;
- видеть запуск и восстановление;
- выполнять безопасную остановку;
- получать технические детали только при необходимости.

Главный принцип:

> Пользователь должен понимать состояние системы без чтения логов и знания внутренней архитектуры.

## 2. Визуальный референс

Файл в корне репозитория:

```text
ChatGPT Image 23 июл. 2026 г., 02_37_00.png
```

является визуальным референсом RadioFactory UI.

Он задаёт:

- тёмный футуристичный стиль;
- профессиональную панель управления;
- высокую информационную плотность без хаоса;
- крупные понятные статусы;
- карточки Station;
- акцент на LIVE-мониторинг;
- визуальное разделение нормальной работы, предупреждений и ошибок.

Референс не является пиксельно точным макетом. При конфликте приоритет имеет документация.

## 3. Целевой пользователь

Интерфейс проектируется для владельца, который не обязан быть программистом.

Не требуется знание systemd, FFmpeg, RTMP, PID, JSON, путей сервера или YouTube API.

Техническая информация доступна вторым уровнем.

## 4. Логическая структура

```text
Dashboard
├── Workspace Overview
├── YouTube Channels
│   └── Stations
├── Station Details
├── Media
├── Events / Notifications
└── Workspace Settings
```

Точная компоновка навигации ещё не утверждена.

## 5. Главная страница

Пользователь должен сразу понять:

- сколько каналов подключено;
- сколько Station существует;
- какие ведут LIVE;
- какие запускаются;
- какие остановлены;
- какие ждут ресурсы;
- какие восстанавливаются;
- где предупреждения;
- где Error;
- есть ли глобальная проблема Workspace.

## 6. Группировка

Station визуально группируются по YouTube Channel.

Интерфейс не должен смешивать понятия:

- YouTube Channel;
- Station;
- Broadcast.

## 7. Карточка Station

Карточка показывает без открытия страницы:

- название Station;
- YouTube Channel;
- основной статус;
- активность LIVE;
- текущий трек;
- этап запуска;
- ожидание ресурса;
- предупреждение;
- Error;
- выполняемую команду;
- быстрые действия.

Для активного LIVE полезны:

- текущий трек;
- прогресс трека;
- прошедшее или оставшееся время LIVE;
- предупреждения.

Точный набор метрик ещё не утверждён.

## 8. Быстрые действия

Подтверждены:

- `Запустить`;
- `Остановить`;
- `Открыть Station`.

Force Stop не должен быть случайной основной кнопкой. Он визуально отделён, требует подтверждения и объясняет последствия.

## 9. Статусы

Смысловые группы:

- остановлена;
- запускается;
- готовит LIVE;
- ждёт ресурс;
- ведёт LIVE;
- штатно останавливается;
- принудительно останавливается;
- восстанавливается;
- предупреждение;
- Error.

Карточка имеет один основной статус. Предупреждение показывается дополнительно.

Цвет не является единственным сигналом: используются текст, иконка и визуальное состояние.

## 10. Детальная страница Station

Подтверждённые области:

- Overview;
- управление LIVE;
- Media;
- Import;
- предупреждения и ошибки;
- History / Events;
- Settings.

Точная система вкладок ещё не утверждена.

## 11. Overview

Overview отвечает:

- работает ли Station;
- что происходит сейчас;
- какой LIVE активен;
- какой трек играет;
- сколько времени прошло;
- есть ли проблема;
- какая команда выполняется;
- что произойдёт дальше.

## 12. Запуск LIVE

После `Запустить` Dashboard:

1. подтверждает приём команды;
2. блокирует дублирующую команду;
3. показывает этап;
4. сообщает об ожидании ресурса;
5. показывает итог.

## 13. Остановка

### Штатная

Интерфейс объясняет:

- текущий трек доиграет;
- новые треки не запустятся;
- LIVE завершится после трека;
- следующий LIVE не запустится.

### Принудительная

Перед исполнением показывается подтверждение:

- LIVE прервётся немедленно;
- текущий трек может оборваться;
- действие может быть необратимым.

Кнопки: `Остановить` и `Отмена`.

## 14. Пошаговый запуск

Нельзя показывать только `Starting`.

Нужно различать:

- проверку готовности;
- подготовку ресурсов;
- создание YouTube LIVE;
- запуск потока;
- подключение;
- подтверждение эфира.

Если этап долгий, интерфейс объясняет, что система работает, а не зависла.

## 15. Waiting States

Отсутствие ресурса показывается как ожидание:

- ждёт музыку;
- ждёт loop-видео;
- ждёт превью;
- ждёт название;
- ждёт описание.

Dashboard называет ресурс, показывает действие и ведёт в нужный раздел Media.

После добавления ресурса подготовка продолжается автоматически.

## 16. Предупреждения

Предупреждение:

- не заменяет основной статус;
- не останавливает Station;
- объясняет последствия;
- рекомендует действие;
- показывает время появления;
- доступно в деталях.

## 17. Ошибки

Первый уровень:

- что произошло;
- что затронуто;
- что система уже сделала;
- работают ли остальные Station;
- что требуется пользователю.

Второй уровень:

- код ошибки;
- время;
- компонент;
- Station;
- Broadcast;
- диагностические сведения;
- связанные события.

## 18. Глобальная авария

Одна общая причина для нескольких Station показывается одной аварией Workspace.

Она содержит:

- описание причины;
- затронутые Station;
- состояние незатронутых;
- действия системы;
- требуемое действие;
- технический код.

Карточки могут ссылаться на общую аварию.

## 19. Восстановление

Пользователь видит:

- что Station восстанавливается;
- продолжается тот же LIVE или создаётся новый;
- текущий этап;
- итог;
- требуется ли действие.

Внутренние попытки не создают поток одинаковых тревог.

## 20. Очередь восстановления

После перезапуска Dashboard показывает:

- ожидающие Station;
- текущую восстанавливаемую Station;
- успешно восстановленные;
- Station в Error;
- продолжение очереди после ошибки одной Station.

## 21. Media

Категории:

- Music;
- Loop Videos;
- Thumbnails;
- Titles;
- Descriptions.

Действия:

- загрузить;
- видеть прогресс;
- отменить;
- искать;
- сортировать;
- просматривать;
- переименовывать;
- удалять.

Обрабатываемые и готовые ресурсы визуально различаются.

## 22. Import

Основной сценарий — Drag & Drop.

Для задания показываются:

- тип ресурса;
- имя или количество файлов;
- этап;
- прогресс;
- успешный результат;
- понятная ошибка;
- безопасная отмена.

## 23. History and Events

Пользовательская история может включать:

- запуск Station;
- создание и начало LIVE;
- завершение;
- восстановление;
- Error;
- штатную остановку;
- Force Stop;
- импорт;
- предупреждение;
- устранение предупреждения.

Полный каталог и срок хранения ещё не утверждены.

## 24. Logs

Логи нужны для диагностики, но не являются главным интерфейсом.

Потенциальные функции:

- просмотр;
- фильтрация по времени;
- фильтрация по Station и Broadcast;
- копирование технических данных.

## 25. Workspace Settings

Глобальные настройки отделены от Station.

Потенциальные области:

- YouTube Channels;
- Telegram;
- часовой пояс;
- язык;
- ресурсные ограничения;
- уведомления;
- обновления;
- backup.

Конкретные настройки ещё не утверждены.

## 26. Актуальность данных

Dashboard обновляет состояние без ручной перезагрузки.

Независимо от технологии он должен:

- показывать потерю связи;
- не выдавать старые данные за актуальные;
- синхронизироваться после восстановления;
- не дублировать команды.

## 27. Жизненный цикл команды

```text
Не отправлена
→ Принята
→ Выполняется
→ Выполнена
```

или

```text
Не отправлена
→ Принята
→ Выполняется
→ Не выполнена
```

Повторное нажатие не создаёт повторный LIVE или конфликтующий процесс.

## 28. Progressive Disclosure

Порядок отображения:

1. понятное состояние;
2. подробности;
3. технические данные и логи.

Пользователь не видит код вместо объяснения, а разработчик не теряет диагностику.
