# Architecture Decisions

Актуальное поведение находится в тематических документах. Здесь хранится история решений.

## №1–11

1. Поддерживаются несколько YouTube-каналов и несколько Station внутри каждого.
2. Главный объект — Workspace.
3. Station автономна и хранит собственные файлы и настройки.
4. Главная страница — Workspace Dashboard со всеми каналами и Station.
5. Station создаётся через мастер, без шаблонов и клонирования.
6. YouTube Channel хранит OAuth, API, плейлисты и обязательные настройки публикации; отсутствующий плейлист создаётся автоматически.
7. Импорт: Drag & Drop, фоновые независимые очереди, прогресс, понятные ошибки и отмена.
8. Медиатека поддерживает поиск, сортировку, просмотр, удаление и переименование.
9. Запуск LIVE пошаговый; между LIVE выполняется изолированный Cleanup.
10. После трёх неудачных попыток останавливается только проблемная Station и отправляется уведомление.
11. Telegram использует уровни, антиспам и группировку общих ошибок.

## №12 — Music Engine

Каждый цикл содержит все треки Station по одному разу в случайном порядке без повторов. Изменение библиотеки влияет только на следующий цикл. Новый LIVE начинает новый цикл. Следующий трек запускается только если полностью помещается в оставшееся время; треки не обрезаются.

## №13 — Random Resource Selection

Каждый LIVE случайно выбирает музыкальный цикл, loop-видео, превью, название и описание. При наличии альтернатив ресурсы не повторяются в двух последовательных LIVE.

## №14 — Pending LIVE Package

До создания LIVE формируется неизменяемый пакет. При ошибке создания используется тот же пакет. Новый пакет формируется после успешного LIVE, отмены или инвалидирующего изменения ресурсов.

## №15 — Continuous Radio

LIVE → естественное завершение → Cleanup → следующий LIVE. Ожидание по расписанию отсутствует.

## №16 — Waiting States

Недостающий ресурс является ожиданием, а не ошибкой. Station продолжает автоматически после появления ресурсов.

## №17 — Восстановление LIVE

Разрыв менее трёх минут — продолжить тот же LIVE. Три минуты и более — завершить старый, выполнить Cleanup и создать новый.

## №18 — Позиция трека

При восстановлении старого LIVE текущий трек продолжается с максимально точной позиции прерывания.

## №19 — Automation First

Сначала используются все безопасные автоматические способы, затем привлекается пользователь.

## №20 — Изоляция аварий

Неудача одной Station переводит только её в Error. Общая причина оформляется как авария Workspace.

## №21 — Два уровня ошибки

Сначала понятное объяснение, затем код и технические сведения.

## №22 — Предупреждения

Некритичная проблема не останавливает Station, показывается в Dashboard и отправляется в Telegram.

## №23 — Перезагрузка

Восстанавливаются только ранее работавшие Station. Ручные остановки сохраняются. Запуск идёт через очередь с учётом ресурсов.

## №24 — Ошибка в очереди

После неудачных попыток Station получает Error и уведомление, а очередь остальных продолжается.

## №25 — Недостаток места

Останавливаются только затронутые Station. После освобождения места выполняется автоматическое восстановление; при неудаче — Error.

## №26 — Две остановки

`Остановить` завершает текущий трек и корректно закрывает LIVE. `Остановить принудительно` требует подтверждения и немедленно завершает LIVE.
